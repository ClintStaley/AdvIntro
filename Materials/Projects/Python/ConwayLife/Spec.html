ConwayLifeBa.py

1. Extend the program to allow one to press one of the number keys 1-4, then click the mouse while the key is depressed, and as a result get one of the 4 canned patterns r-pentomino(1), glider(2), spaceship(3), or glider gun(4). Place the center of the pattern at the mouse click, wrapping around the edges toroidally if needed. Do a full copy of the rectangle of the pattern, thus also clearing any other cells than those in the pattern within its rectangle.

And, *use the list provided* at the bottom of the ConwayLifeAns.py file for the pattern contents; don't write all that by hand!

Note that there is a pygame.MOUSEBUTTONUP event to match the MOUSEBUTTONDOWN event. This will let you know if and when keys are released, which might be part of your logic for this to work. (There are approaches that don't need the MOUSEBUTTONUP, and that's fine if you use them)

2. Track the number of generations for which each living cell has been alive. Increase the cell's darkness by 1/4 for each generation it's been alive, so that it gradually changes from light gray to fully black over 4 generations. Do this by changing at most 7 lines of code in the program, and be prepared to describe those 7 lines in your demo.

3. Add handling to the **numpad, not keyboard** + and - keys that will, respectively, speed the simulation up or slow it down by a factor of 2. See how fast you can run the simulation before it "tops out" at a given frame rate due to compute limitations. The constants for those two keys are K_KP_PLUS and K_KP_MINUS. The keyboard + is harder to use because you have to separately track the shift operation.

Gold

ConwayLifeGo.py

Some cells change in each generation, and when you think abou it, only their *neighbors* can possibly change in the following generation, since only their neighbors have new neighbor counts. If we keep a list of cells whose neighbors have changed, we only need to reexamine those in the following generation. We can maintain such a list by adding the neighbors of any cell who changes in the current generation. (Note that even a changed cell itself will not re-change in the next generation unless one of its neighbors has changed.)

Track those cells, and reexamine only them in each generation, using a single loop to go through them -- no nested row/col loop.

Also, add a generation count, displayed in the title bar, so you can track the number of generations per second you're able to run.

And, for your own testing and for the demo, add a print of the size of that neighbor-changed list on each generation. This may float up or down a bit, but should not grow to a sizeable portion of the cells. And, for a simple pattern like a 2x2, which is stable and nonchanging, that neighbor-changed count should drop to 0.

Specifically, check the following:

1. You have *no* nested for-loops in method "step", just a single loop through the neighbors-changed list and possible adjustment of only those cells. You may find you want a deep-copy of the grid, however. That's OK.
